## Executive Summary

Synced 3,640 ET documents (fuente 116 = Entrada Producto Terminado) from SAG into the universal ProductionEvent model. This is the **first real data** written to ProductionEvent, validating that the ERP-agnostic model works correctly with production documents.

**Critical discovery:** ET documents in SAG PYA are **header-only**. They carry zero line items in MOVIMIENTOS_ITEMS. This is not a bug in Agentik's query — it's how SAG PYA stores ET documents. ET headers are auto-generated by the production system and contain a cross-reference (`ss_remision`) back to the originating OP.

The model handled this gracefully: `referenceCode: null`, `lineCount: 0`, `quantity: 0` — exactly as designed in PRODUCTION-EVENT-MODEL-REVIEW-01 for header-only events.

### Why 0 lines?

**Confirmed: ET is header-only in SAG PYA.**

Evidence:

| Test | Query | Result |
|---|---|---|
| Direct items query (no JOIN) | `SELECT * FROM MOVIMIENTOS_ITEMS WHERE ka_nl_movimiento = {et_id}` | 0 rows |
| Count all ET items | `SELECT COUNT(*) FROM MOVIMIENTOS_ITEMS mi JOIN MOVIMIENTOS m ON ... WHERE m.ka_ni_fuente = 116` | 0 |
| Individual header check (5 samples) | Per-header COUNT query | 0 each |
| OP comparison (same query) | `SELECT COUNT(*) FROM MOVIMIENTOS_ITEMS WHERE ka_nl_movimiento = {op_id}` | 8 items |

**Root cause:** SAG PYA's production module generates ET documents automatically as header-level events. The `sv_observaciones` field confirms: `"Generado por el sistema de Produccion. Entradas a Almacen de Orden de Produccion"`. The system records THAT a production entry happened (header), but the item details live in the originating CN/OP documents, not in ET's MOVIMIENTOS_ITEMS.

**This is NOT a bug.** The ProductionEvent model was designed to handle this case (PRODUCTION-EVENT-MODEL-REVIEW-01, Phase 3 — "ET with 0 lines: header only — SAG anomaly: Supported? YES").

## Phase 7 — Analisis Forense

### Finding 1: ET is header-only in SAG PYA

0% of ET documents have lines in MOVIMIENTOS_ITEMS. This is confirmed across:
- 8 batches of item queries (all returned 0)
- Direct per-header queries
- COUNT aggregate query

The problem is NOT in Agentik. ET documents in SAG PYA's production module simply don't generate line items.

### Finding 2: ET carries OP cross-reference

100% (3,640/3,640) of ET headers have `ss_remision` with format `"{OP_number}-{sequence}"` (e.g. "3318-1"). This is the link to the originating ProductionOrder.

This means ET CAN be cross-referenced with OP to determine which references were completed, even without ET lines:

```
ET.ss_remision = "3318-1"
  → OP #3318
    → ProductionOrderLine.referenceCode = "L-4501234"
    → sizes, colors, quantities from OP lines
```

### Finding 3: ET documents are auto-generated

The `sv_observaciones` field says: "Generado por el sistema de Produccion. Entradas a Almacen de Orden de Produccion". This means ET is created automatically when production is completed — not manually by an operator.

### Finding 4: ET date range spans full history

| Metric | Value |
|---|---|
| Earliest ET | #1 — 2020-11-19 |
| Latest ET | #3658 — 2026-06-26 |
| Date span | 5 years, 7 months |
| Active | YES — latest is 3 days ago |

### Finding 5: ET does NOT represent partial completion

ET represents completion of an entire OP (or OP section). The `ss_remision` pattern `{OP}-{seq}` suggests that a single OP can generate multiple ET entries (different sequences for different completion batches).

### What ET does NOT enable (needs CN)

| Capability | Status | Why |
|---|---|---|
| Per-reference completion tracking | BLOCKED | ET has no lines; needs CN for reference detail |
| Per-variant completion | BLOCKED | ET has no size/color; needs CN or OP cross-ref |
| Quantity tracking | BLOCKED | ET quantity = 0 (header-only) |
| Full lifecycle timeline | BLOCKED | Needs CN to know when production started |

### How to unlock per-reference tracking

The `ss_remision` field (format: `{OP_number}-{sequence}`) can be cross-referenced against ProductionOrder:

```
1. Parse OP number from ET.productionOrderRef (e.g. "3318-1" → OP #3318)
2. Find ProductionOrder where documentNumber = "3318"
3. Get ProductionOrderLine references for that OP
4. These references are the ones completed by this ET
```

This cross-reference is NOT implemented in this sprint (it belongs in PRODUCTION-LIFECYCLE-01 or PRODUCTION-STAGE-ACTIVATION-01).
