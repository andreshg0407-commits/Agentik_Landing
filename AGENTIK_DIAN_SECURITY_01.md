## PKCS#12 Parsing (`pkcs12-parser.ts`)

Uses `node-forge` to extract signing material from a `.p12` file.

```typescript
const parsed = parsePkcs12(p12Buffer, password, alias?);
// parsed.privateKeyPem  — RSA private key PEM (for createSign())
// parsed.certDer        — DER-encoded certificate (for BinarySecurityToken)
// parsed.commonName     — Certificate Subject CN
// parsed.validUntil     — Certificate expiry date
// parsed.validFrom      — Certificate issuance date
```

Supports:
- `pkcs8ShroudedKeyBag` (most common DIAN certs)
- `keyBag` fallback for older P12 structures
- Optional alias selection for multi-cert keystores

## WS-Security Signing (`xml-signer.ts`)

Implements OASIS WS-Security XML-DSIG as required by DIAN guide Section 2.6.

### Algorithm specification

| Component | Value |
|-----------|-------|
| Signature algorithm | `RSA-SHA256` |
| Canonicalization | Exclusive C14N (`http://www.w3.org/2001/10/xml-exc-c14n#`) |
| Digest algorithm | `SHA-256` |
| Key reference | `BinarySecurityToken` (X509v3) |
| Signed elements | `u:Timestamp`, `a:To` |

### Exclusive C14N approach

Rather than a full general-purpose C14N parser, we generate each element to be
signed in canonical form directly. This is correct because:

1. All XML is generated by us (controlled input, no third-party XML)
2. Namespace prefixes are known and constant at build time
3. Elements have no inherited namespace ambiguity
4. Attributes are sorted alphabetically at generation time

**Timestamp canonical form:**
```xml
<u:Timestamp xmlns:u="http://docs.oasis-open.org/wss/2004/01/oasis-200401-wss-wssecurity-utility-1.0.xsd" u:Id="_ts_xxx"><u:Created>2026-05-10T...</u:Created><u:Expires>2026-05-10T...</u:Expires></u:Timestamp>
```

**To canonical form:**
```xml
<a:To xmlns:a="http://www.w3.org/2005/08/addressing" xmlns:u="http://docs.oasis-open.org/wss/2004/01/oasis-200401-wss-wssecurity-utility-1.0.xsd" u:Id="_to_xxx">https://vpfe-hab.dian.gov.co/...</a:To>
```

Both have namespace declarations sorted alphabetically (`xmlns:a` before `xmlns:u`),
matching exactly what a receiver's Exclusive C14N would produce.

### Signing flow

```
1. SHA-256(C14N(Timestamp)) → DigestValue for Timestamp Reference
2. SHA-256(C14N(To)) → DigestValue for To Reference
3. Build SignedInfo in canonical form (compact, no whitespace)
4. RSA-SHA256(C14N(SignedInfo)) → SignatureValue
5. Return BinarySecurityToken XML + ds:Signature XML
```

### SOAP Security header result

```xml
<o:Security xmlns:o="..." xmlns:u="..." soap:mustUnderstand="1">
  <u:Timestamp xmlns:u="..." u:Id="_ts_xxx">...</u:Timestamp>
  <o:BinarySecurityToken u:Id="_bst_xxx" ValueType="...X509v3" EncodingType="...Base64Binary">
    BASE64_DER_CERT
  </o:BinarySecurityToken>
  <ds:Signature xmlns:ds="http://www.w3.org/2000/09/xmldsig#">
    <ds:SignedInfo>...</ds:SignedInfo>
    <ds:SignatureValue>BASE64_SIGNATURE</ds:SignatureValue>
    <ds:KeyInfo>
      <o:SecurityTokenReference>
        <o:Reference URI="#_bst_xxx" ValueType="...X509v3"/>
      </o:SecurityTokenReference>
    </ds:KeyInfo>
  </ds:Signature>
</o:Security>
```

## DianClient Full Flow

`getAcquirer()` now implements the complete request lifecycle:

```
1. validateCertificateConfig()   — path + password + file existence
2. loadCertificateBuffer()       — read .p12 bytes from filesystem
3. parseCertificateFromBuffer()  — node-forge PKCS#12 parse → private key + DER cert
4. buildGetAcquirerRequest()     — SOAP envelope + signing inputs
5. signWssElements()             — RSA-SHA256 + Exclusive C14N → BST + Signature XML
6. injectSignatureIntoEnvelope() — replace placeholders with signed XML
7. dispatch()                    — fetch() with timeout + 1 retry on 5xx
8. validateSoapResponse()        — detect SOAP Fault before parsing
9. parseGetAcquirerResponse()    — typed response
```

HTTP dispatch rules:
- Timeout: `DIAN_TIMEOUT_MS` (default 30s, max 60s)
- Retry once on HTTP 5xx or AbortError (network timeout)
- No retry on 4xx (auth / bad request — retrying won't help)
- Never log `signedXml` in production (contains WS-Security tokens)

## SOAP Builder Updates

### `BuiltSoapRequest.signingInputs`

`buildGetAcquirerRequest()` now returns `signingInputs` alongside `xmlBody`:

```typescript
const built = buildGetAcquirerRequest(options);
// built.xmlBody        — unsigned envelope with placeholder comments
// built.signingInputs  — {timestampId, timestampCanonical, toId, toCanonical}
```

### `injectSignatureIntoEnvelope()`

Replaces placeholder comments in the unsigned envelope:

```typescript
const signedXml = injectSignatureIntoEnvelope(built.xmlBody, bstXml, signatureXml);
// Replaces:
//   <!-- __DIAN_BST_PLACEHOLDER__ -->  → BinarySecurityToken element
//   <!-- __DIAN_SIG_PLACEHOLDER__ -->  → ds:Signature element
```

### `buildWsaHeaders()` — `toId` param

The `a:To` element now accepts an optional `u:Id` for signing:

```xml
<a:To u:Id="_to_xxx">https://vpfe-hab.dian.gov.co/WcfDianCustomerServices.svc</a:To>
```

The `xmlns:u` is inherited from the root `<soap:Envelope>` (declared via `DIAN_SOAP_NAMESPACES`).

## Tenant Loader: SecureVault Integration

`loadTenantDianContext()` now checks the `secretsJson` version before selecting
the password retrieval path:

```
version "2" → SecureVault.readSecret(dianCertRef(orgId, certId), envelope, context)
version "1" → legacy certificate-vault.ts FilesystemCertificateVault (backward compat)
other       → legacy path
```

This enables gradual migration: tenants are upgraded to v2 as they're onboarded with
`SecureVault.writeSecret()`. Existing v1 tenants continue to work unchanged.

## Security Audit

| Risk | Mitigation |
|------|-----------|
| Private key in logs | `parseCertificateFromBuffer()` and `signWssElements()` never log key material |
| Signed XML in logs | `dispatch()` never logs `signedXml` (contains WS-Security tokens) |
| Wrong cert password | `parsePkcs12()` throws `DianCertificateError` immediately — no partial state |
| Tampered SOAP response | `validateSoapResponse()` + SOAP Fault detection before response parsing |
| Request replay | Timestamp TTL 60s enforced by DIAN server |
| Environment mixing | `loadTenantDianContext()` validates env match before returning context |
| Cross-tenant cert | SecureVault checks organizationId match; Prisma queries filter by orgId |
